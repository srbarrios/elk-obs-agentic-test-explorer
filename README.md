# 🤖 Elastic Observability Agentic Test Explorer

An autonomous, multi-agent Quality Assurance (QA) framework designed to intelligently explore, test, and validate the Elastic Observability UI. 

Powered by **LangGraph**, **LangChain**, **Playwright**, and **Google Gemini 3.1**, this framework doesn't just run static tests; it reads documentation, visually validates charts, self-heals from UI errors, captures screenshot evidence, and writes its own Markdown test reports.


## ✨ Key Features

* **Multi-Agent Swarm:** Uses a Supervisor routing model to distribute tasks among five highly specialized AI personas (Logs, APM, Metrics, Synthetics, Alerting).
* **Self-Healing Browser Execution:** Playwright actions are wrapped in a custom self-healing loop. If a button is missing or a selector changes, the agent receives the error, re-evaluates the DOM, and tries a new strategy instead of crashing.
* **Visual Validation:** Agents can take screenshots of complex Canvas/SVG elements (like Service Maps) and use Gemini Vision to analyze them for rendering anomalies.
* **MCP Documentation Integration:** Agents proactively query the Elastic Docs via the Model Context Protocol (MCP) to learn UI paths *before* executing actions.
* **Automated Artifact Generation:** Every test mission automatically generates an isolated folder containing raw execution traces, bug screenshots, and a beautiful executive Markdown report.

## 📂 Project Structure

* `main.py`: The core orchestrator. Handles CLI arguments, persistent memory, browser initialization, and report generation.
* `agents.py`: Contains the LangGraph state definition, the Supervisor routing logic, and the specialized QA agent personas.
* `custom_tools.py`: The tool factory. Contains visual validation tools, bug screenshot tools, and Elastic MCP/Skill connectors.
* `auth_setup.py`: A utility script to perform initial login and save the session state to bypass login screens during testing.
* `missions.yaml` *(User Created)*: The configuration file defining the test scenarios.

## 🚀 Getting Started

### 1. Prerequisites
Ensure you have Python 3.11+ installed. Install the required dependencies:
```bash
pip install langchain langchain-core langchain-community langgraph langchain-google-genai playwright pillow pyyaml python-dotenv langchain-mcp-adapters
```
Install the Playwright Chromium browser:
```bash
playwright install chromium
```

### 2. Environment Configuration
Create a `.env` file in the root directory and add your API keys and Kibana environment details:
```env
GOOGLE_API_KEY="your_gemini_api_key_here"
KIBANA_URL="http://localhost:5601"
KIBANA_USERNAME="elastic"
KIBANA_PASSWORD="changeme"
```

### 3. Authentication
Run the authentication setup script. This will open a browser, log into Kibana using your `.env` credentials, and save the session to `auth.json`. This allows the agents to run tests without needing to log in every time.
```bash
python auth_setup.py
```

### 4. Defining Missions
Create a `missions.yaml` file defining the specific UI tests. Each mission needs a unique `thread_id` (for persistent memory) and a `prompt`.
```yaml
missions:
  - thread_id: "obs_logs_exploration_01"
    prompt: >
      Navigate to Observability Logs. Perform a KQL search for 'error'. 
      Verify the detail flyout renders correctly and highlights the search term.
```

## 🎮 Running the Framework

Execute your test suite by pointing `main.py` to your missions file:

```bash
python main.py --missions missions.yaml
```

### CLI Arguments

* `--missions <file.yaml>`: **(Required)** Path to your test definitions.
* `--headed`: Opens the browser visibly so you can watch the agents perform their tasks (runs headless by default).
* `--clear-memory`: Wipes the SQLite database before running. Highly recommended when iterating on tests to ensure agents don't use memory from previous failed runs.

**Example: Debugging a test run from scratch:**
```bash
python main.py --missions missions.yaml --headed --clear-memory
```

## 📊 Test Artifacts

For every mission executed, the framework generates a dedicated `report_<thread_id>` directory. Inside, you will find:
1.  `traces.log`: A complete, human-readable audit trail of every thought, plan, and tool invocation the agent performed.
2.  `test_report.md`: A concise executive summary generated by the AI detailing the objective, actions taken, bugs found, and a final PASS/FAIL status.
3.  `/screenshots/`: (If applicable) High-resolution image evidence of any UI bugs, missing elements, or visual anomalies discovered by the agents.
